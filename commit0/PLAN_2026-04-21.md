## The "value-add over LLM" framework

For every (architecture Ã— model Ã— library) cell, we compute:

```

value_add_$_pp   = arch_cost / max(value_add_pp, 0.01)        # $ per percentage-point gained
llm_lean         = arch_cost / single_shot_LLM_cost (same model, same lib)

```

This separates **real architectural value** from **spent more LLM money**. A baseline that wins by burning 10Ã— the tokens isn't an architectural win; it's a budget choice.

## Architectural-weakness fingerprints we'll extract

For **OH (agent-loop with tools)**, weakness signature:
- Tokens-per-call rising over the run â†’ conversation history growing without caching
- Cost-per-resolved blowing past `model_input_rate Ã— max_iter` â†’ trial-and-error, not directed search
- Unresolved-with-many-calls â†’ no termination heuristic when stuck
- 0% cache hit rate â†’ "send more tokens", not "be smarter about tokens"

For **KD (per-file decompose + grounding)**, weakness signature:
- `files_accepted < files_total` with no further retry success â†’ some files irrecoverably broken
- `output_tokens > input_tokens` by large margin â†’ regenerating whole files for small changes
- Negative value-add on B2-strong libs â†’ per-file regen damaging working code
- Zero-from-zero on cross-import libs â†’ no test-side visibility
