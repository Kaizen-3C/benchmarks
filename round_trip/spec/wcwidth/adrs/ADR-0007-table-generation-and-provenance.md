# ADR-0007: Unicode Table Generation and Provenance

## Status
Accepted

## Context
The lookup tables (`WIDE_EASTASIAN`, `ZERO_WIDTH`, `VS16_NARROW_TO_WIDE`) are derived from Unicode Consortium data files. They must be reproducible and auditable.

## Decision
Tables are generated by `bin/update-tables.py` using Jinja2 templates in `code_templates/`. The script:

1. Downloads Unicode data files from `unicode.org` (using HTTP with retry logic: up to 6 retries, backoff factor 0.1, retry on HTTP 500/502/503/504).
2. Fetches: `DerivedAge.txt`, per-version `EastAsianWidth-{version}.txt`, per-version `DerivedGeneralCategory-{version}.txt`, and emoji files.
3. Wide table logic:
   - Start with codepoints having EastAsianWidth category `W` or `F`.
   - Subtract codepoints also classified as `Mn`, `Mc`, `Me`, `Cf`, `Zl`, or `Zp` in DerivedGeneralCategory.
   - Subtract Hangul Jamo vowels/trailing consonants: `U+1160–U+11FF` and `U+D7B0–U+D7FF`.
   - Add codepoints in `Sk` category that contain "FULLWIDTH" in their comment (width=2).
   - `Sk` codepoints containing "EMOJI MODIFIER" are classified width=0; remaining `Sk` are width=1.
4. Zero table logic: codepoints with DerivedGeneralCategory `Me`, `Mn`, `Mc`, `Cf`, `Zl`, or `Zp`; plus NULL (U+0000); plus the Hangul Jamo ranges above.
5. VS16 table: parsed from `emoji-variation-sequences.txt` for codepoints preceding `FE0F`. A legacy file from emoji version 12.0 is also merged. Only codepoints not already in the wide table are retained. A single table entry keyed at `"9.0.0"` is produced.
6. Generated files are written to `wcwidth/table_wide.py`, `wcwidth/table_zero.py`, `wcwidth/table_vs16.py`, `wcwidth/unicode_versions.py`, and `docs/unicode_version.rst`.

Downloaded data files are cached locally in `data/`. Files are re-fetched only when the remote `Last-Modified` header is newer than the local file, unless `--no-check-last-modified` is passed.

## Consequences
- Tables are static Python source files committed to the repository.
- Reproducibility requires network access to `unicode.org`.
- The Hangul Jamo zero-width subtraction is a deliberate deviation from raw Unicode data to handle conjoining Jamo correctly.
